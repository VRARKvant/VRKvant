# Roadmap: Внедрение тестового покрытия для VR/AR Kvantum Knowledge Hub

Данная дорожная карта разбита на этапы от базовой настройки до полной автоматизации в CI/CD.

## Этап 1: Подготовка инфраструктуры (Setup)
*Цель: Установить необходимые инструменты и настроить окружение.*

- [ ] **Выбор стека тестирования**: 
  - Юнит/Интеграционные: `Vitest` + `jsdom` (быстрее и современнее Jest, отлично работает с ES-модулями).
  - E2E: `Playwright` (надежная имитация браузера).
- [ ] **Установка зависимостей**:
  `npm install -D vitest jsdom @playwright/test`
- [ ] **Настройка конфигурации**:
  - Создать `vitest.config.js` (указать среду `jsdom`).
  - Инициализировать Playwright: `npx playwright install`.
- [ ] **Обновление `package.json`**:
  - Добавить скрипты: `"test": "vitest run"`, `"test:watch": "vitest"`, `"test:e2e": "playwright test"`.

## Этап 2: Unit-тестирование (Чистые функции и логика)
*Цель: Покрыть тестами независимые модули обработки данных и API.*

- [ ] **Тестирование `js/router.js` (Парсер Frontmatter)**
  - Написать тест для `parseFrontmatter`: проверка корректного извлечения `title`, `module` и т.д.
  - Проверка поведения при отсутствии Frontmatter (возврат чистого контента).
  - Проверка устойчивости к лишним пробелам и спецсимволам.
- [ ] **Тестирование `js/main.js` (Транслитерация)**
  - Написать тест для `autoTransliterate` (и `cyrillicToLatinMap`).
  - Проверить краевые случаи: цифры, спецсимволы, заглавные буквы, длинные пробелы.
- [ ] **Тестирование `js/ui.js` (Группировка)**
  - Написать тест для `groupLessonsByModule`.
  - Убедиться, что уроки без модуля падают в категорию "Разное".
- [ ] **Тестирование `js/api.js` (Сетевые запросы)**
  - Настроить мок (mock) для `fetch` (через `vi.stubGlobal('fetch', ...)`).
  - Тест успешной загрузки `loadPortfolio()`, `loadTracks()`, `loadCheats()`.
  - Тест обработки ошибок: имитация ответа 404 или 500, проверка возврата пустого массива `[]`.

## Этап 3: Интеграционное тестирование (Работа с DOM и State)
*Цель: Убедиться, что UI-функции корректно генерируют HTML, а стейт обновляется.*

- [ ] **Тестирование Store (`js/store.js`)**
  - Создать тесты для проверки подписок (`subscribe`) и обновления (`updateState`).
  - Убедиться, что `Proxy` корректно перехватывает изменения базовых свойств (например, `isDark`).
- [ ] **Тестирование генераторов UI (`js/ui.js`) с использованием JSDOM**
  - Подготовить mock-данные (массив фейковых проектов/треков).
  - Тест `renderPortfolio()`: вызвать функцию и проверить наличие созданных карточек (`.card-link`) в моковом контейнере `document.body`.
  - Тест `buildLeftSidebar()`: проверить корректную вложенность треков и уроков.
  - Тест `buildToC()`: передать сгенерированный HTML с тегами `<h2>` и `<h3>`, проверить сборку оглавления.

## Этап 4: End-to-End (E2E) тестирование критических сценариев
*Цель: Проверить реальный пользовательский опыт в браузере с помощью Playwright.*

- [ ] **Сценарий 1: Роутинг и навигация**
  - Открыть главную страницу (`/#home`).
  - Кликнуть на вкладку "Треки" в навигации.
  - Убедиться, что URL изменился на `/#tracks`, а блок `#tracks` получил класс `active`.
  - Кликнуть по карточке урока, дождаться появления заголовка `<h1>` статьи.
- [ ] **Сценарий 2: Переключение темы**
  - Найти кнопку смены темы (`#btn-toggle-theme`).
  - Кликнуть и проверить добавление класса `dark` к тегу `<html>`.
  - Проверить сохранение стейта в `localStorage`.
- [ ] **Сценарий 3: Редактор Markdown**
  - Перейти по роуту `/#editor` (или использовать шорткат Ctrl+Shift+E).
  - Ввести текст в `<textarea id="markdown-input">`.
  - Кликнуть на тулбар (например, `toolbar-bold`).
  - Проверить, что превью (`#editor-preview`) корректно отрендерило теги `<strong>`.

## Этап 5: Внедрение в CI/CD (GitHub Actions)
*Цель: Запретить публикацию сломанного кода.*

- [ ] **Обновление `.github/workflows/build.yml`**
  - Добавить шаг запуска тестов перед билдом:
    ```yaml
    - name: Install dependencies
      run: npm install

    - name: Run Tests (Unit & Integration)
      run: npm run test

    - name: Run E2E Tests
      run: npx playwright test
      
    - name: Build CSS
      run: npm run build:css
    ```
- [ ] Настроить блокировку слияния (Merge) в ветку `main`, если тесты (Actions) завершились с ошибкой.

# Roadmap: Прокачка редактора статей и оптимизация Sveltia CMS

Этот план охватывает внедрение умного редактора метаданных, горячих клавиш, 1-в-1 превью, а также исправление архитектурных недочетов CMS, выявленных ранее.

## Этап 1: Оптимизация и безопасность CMS (Конфигурация)
*Цель: Закрыть дыры в безопасности авторизации и сделать сохранение файлов предсказуемым.*

- [ ] **Безопасность OAuth (`api/callback.js`)**:
  - Заменить `window.opener.postMessage(..., "*")` на жесткую проверку `e.origin`.
  - Разрешить отправку токена только на белые адреса: `https://vrarkvant.github.io` и `http://localhost:3000` (для локальной разработки).
- [ ] **Форматы файлов (`admin/config.yml`)**:
  - В каждую коллекцию (`blender`, `unity`, `unreal`, `portfolio`, `blog`) явно добавить параметры:
    `extension: "md"`
    `format: "yaml-frontmatter"`
  - *Причина: Это гарантирует, что CMS не сломает ваш кастомный парсер из `js/router.js`.*
- [ ] **Решение проблемы с `base_url`**:
  - Продумать процесс локальной разработки: создать инструкцию для команды, чтобы локально они временно меняли `base_url` в конфиге, либо настроить Vercel CLI для проксирования локальных запросов.
- [ ] **Опционально (DRY конфига)**: Оценить возможность слияния коллекций треков (`blender`, `unity`, `unreal`) в одну общую коллекцию `tracks` с использованием динамического параметра пути `path: "{{track_name}}/{{slug}}"`, чтобы не дублировать поля.

## Этап 2: Редизайн UI панели метаданных (HTML & CSS)
*Цель: Сделать интерфейс настроек статьи компактным и визуально понятным.*

- [ ] **Сворачиваемая панель**: 
  - Обернуть инпуты метаданных в тег `<details class="group ...">` с настраиваемым тегом `<summary>`.
- [ ] **Умный выбор модуля**: 
  - Заменить обычный `<input>` для модуля на связку `<input list="modules-list">` и пустой `<datalist id="modules-list">`.
- [ ] **Блок авто-нумерации**:
  - Добавить кнопку `<button id="btn-auto-order">` рядом с полем ввода номера урока (Order).
- [ ] **Визуальный Color Picker**:
  - Скрыть инпут `#meta-color`.
  - Создать блок с круглыми `<input type="radio" class="peer sr-only">` и стилизовать их через классы `peer-checked:ring-4`.
- [ ] **Тулбар редактора (Цитаты)**:
  - Добавить кнопку-выпадашку (dropdown) для цитат с пунктами "Важно (Warn)" и "Заметка (Tip)", стилизованную через Tailwind (`group-hover:block`).

## Этап 3: JS-Автоматизация Метаданных (`js/main.js` / `js/router.js`)
*Цель: Заставить UI-элементы работать с данными из вашего глобального Store.*

- [ ] **Автозаполнение модулей**:
  - Написать функцию `populateModulesList()`, которая берет текущий трек из `store.tracks`, собирает уникальные модули (`[...new Set(...)]`) и генерирует `<option>` внутри `#modules-list`.
- [ ] **Логика авто-нумерации урока**:
  - Повесить обработчик клика на `#btn-auto-order`.
  - Логика: найти все уроки выбранного трека/модуля, найти `Math.max(order)` и подставить в инпут значение `max + 10` (оставляя шаг для вставки промежуточных уроков).
- [ ] **Реактивность Color Picker'а**:
  - Настроить слушатель на радио-кнопки цветов, чтобы при клике их значение (`value`) передавалось в скрытый инпут и сразу вызывалась функция `updateCardPreview()`.

## Этап 4: Прокачка Markdown-редактора (`js/main.js`)
*Цель: Ускорить написание статей с помощью автодополнения и горячих клавиш.*

- [ ] **Перехват нажатий клавиш (`keydown`)**:
  - Повесить слушатель на `#markdown-input`.
- [ ] **Автозакрытие скобок**:
  - Если нажата открывающая скобка `( { [ " ' \``, автоматически вставлять закрывающую и помещать курсор между ними (с сохранением выделенного текста, если он был).
- [ ] **Горячие клавиши (Shortcuts)**:
  - Добавить перехват `Ctrl/Cmd + B` для оборачивания выделенного текста в `**`.
  - Добавить перехват `Ctrl/Cmd + I` для оборачивания в `*`.
  - *Обязательно вызывать `dispatchEvent(new Event('input'))` после изменений, чтобы превью обновилось.*
- [ ] **Шаблоны цитат**:
  - Обновить функцию `insertTemplate(type)`: для `quote-warn` вставлять шаблон `> [!WARNING]`, для `quote-tip` — `> [!TIP]`.

## Этап 5: Синхронизация 1-в-1 Превью (`js/ui.js` / `js/main.js`)
*Цель: Разработчик должен видеть статью и карточку точно так же, как ее увидит пользователь.*

- [ ] **Стилизация превью статьи**:
  - Убедиться, что контейнер `#editor-preview` имеет идентичные CSS-классы с `#article-content` (шрифты, отступы, поддержка темной темы `dark:text-white`).
- [ ] **Вызов парсера**:
  - Проверить, что внутри `updateEditorPreview` вызываются функции-декораторы: `makeHeadersCollapsible()`, `styleSpecialQuotes()`, `addCodeFeatures()`.
- [ ] **Живое превью карточки трека/проекта**:
  - Убедиться, что `updateCardPreview` собирает HTML карточки *точно* с такими же Tailwind-классами, как функции `renderPortfolio` и `renderHomeTracks`.
