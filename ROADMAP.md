# Завершенные этапы (Текущий функционал)
* **✅ Умный поиск (Fuse.js)**: Реализован нечеткий полнотекстовый поиск по всей базе знаний. Скрипт сборки генерирует единый индекс `search_index.json`, а клиентская часть (через модальное окно) мгновенно ищет статьи даже с опечатками.
* **✅ Обработка метаданных (Frontmatter)**: Настроен парсинг YAML-фронтматтера из `.md` файлов. Теперь статьи автоматически получают авторов, дату публикации, теги и категорию (выводится в начале статьи).
* **✅ Интерактивный Граф знаний**: Внедрен ForceGraph. Скрипт сборки сканирует Markdown-файлы на наличие внутренних связей и генерирует `graph.json` для визуализации паутины.
* **✅ Автогенерация манифестов**: Node.js скрипт (`build.js`) самостоятельно сканирует директорию `/articles` и строит `tracks.json`, `cheats.json` и `portfolio.json`.
* **✅ Динамический UI**: Внедрено правое меню с вкладками (Оглавление / Ссылки / Локальный граф), кастомные теги `[gallery:]` и `[compare:]`.

# Архитектурные замечания и развитие (Архитектура 3.0)

Текущая архитектура (Vanilla JS + Hash Routing + Node.js Build) отлично справляется с задачей создания быстрой и легкой статической базы знаний. Однако по мере роста проекта стоит обратить внимание на следующие точки роста:

### 1. Отказ от Tailwind CDN в Production
**Проблема:** Использование `<script src="https://cdn.tailwindcss.com"></script>` в `index.html` замедляет первую загрузку страницы и заставляет браузер клиента компилировать CSS на лету. Это плохая практика для Production.
**Решение:** Настроить сборку Tailwind CLI или PostCSS (в рамках `build.yml` или локального `npm run build`), чтобы генерировать статичный минифицированный `style.css`.

### 2. Кэширование и PWA (Офлайн-режим)
**Проблема:** Каждый переход по статье делает новый `fetch` запрос к `.md` файлу и `.json` манифестам. При плохом интернете интерфейс может казаться "залипающим", так как нет прелоадеров перехода (кроме спиннера внутри контейнера).
**Решение:** * Внедрить Service Worker для кэширования статики, `.json` манифестов и уже прочитанных `.md` статей.
* Превратить сайт в PWA, чтобы студенты могли устанавливать базу знаний как приложение на телефон/ПК и читать статьи в офлайне.

### 3. State Management и Инкапсуляция
**Проблема:** Данные хранятся в глобальном объекте `window.siteData` и DOM-дереве (data-атрибуты). Роутер напрямую манипулирует классами (`classList.add('hidden')`), что делает код хрупким при изменениях верстки.
**Решение:**
* Инкапсулировать состояние в модулях или использовать простую реактивную обертку (например, Proxy).
* Вынести функции рендера в отдельные компоненты, чтобы роутер занимался только вызовом методов "показать представление", а не дергал DOM напрямую.

### 4. Оптимизация парсинга Markdown
**Проблема:** `marked.parse` и кастомные регулярные выражения (gallery, compare) запускаются на клиенте при каждом открытии статьи. Для тяжелых статей (особенно с подсветкой кода) это может вызывать микро-фризы на мобильных устройствах.
**Решение:** * Частично перенести парсинг на этап сборки (Node.js скрипт). Скрипт может генерировать готовые `.html` куски или AST-деревья, оставляя браузеру только вставку готовой разметки в DOM.

### 5. Type Safety (JSDoc / TypeScript)
**Проблема:** Проект полностью написан на Vanilla JS. При добавлении сложных фич (поиск, граф, роутинг) легко допустить ошибки при передаче объектов между файлами.
**Решение:** Начать аннотировать критичные участки кода с помощью JSDoc для получения автодополнения в VS Code, или постепенно перевести ядро логики на TypeScript.
